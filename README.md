Анализ очередей на кассах с использованием SQL

О проекте

В проекте выполнен анализ кассовых операций, чтобы определить периоды, в которые на кассах, вероятно, присутствовала очередь.

Для анализа использовались данные по операциям на кассах и справочник магазинов. Очередь определялась косвенно — через минимальный разрыв между соседними операциями на одной кассе.

Используемые данные

В проекте использовались две таблицы:

- `store_checkout_queues` — данные по кассовым операциям
- `store_stores` — справочник магазинов

Полный датасет не добавлен в репозиторий из-за большого объёма. Для демонстрации структуры данных и логики анализа приложен файл `sample_data.csv`.

Что было сделано

В ходе работы:

- объединены таблицы `store_checkout_queues` и `store_stores` по полю `store_uuid`
- выполнена фильтрация по выбранным магазинам
- приведены типы данных, включая дату и время операций
- рассчитан разрыв между соседними операциями на одной кассе
- выделены и объединены последовательные интервалы с минимальным разрывом
- получены периоды предполагаемой очереди

Логика определения очереди

Очередь считалась присутствующей, если следующая операция на той же кассе начиналась не позднее чем через 5 секунд после завершения предыдущей.

Условие:

`gap_to_next_start_s <= 5`

Такой подход позволяет выделить периоды высокой загрузки кассы и использовать их как прокси для определения очереди.

Результат

Для каждого периода очереди были рассчитаны:

- `queue_start_dt` — время начала периода
- `queue_end_dt` — время окончания периода
- `checks_with_queue` — количество чеков в периоде
- `queue_duration_s` — длительность периода в секундах

SQL-реализация

В файле `solution.sql` содержатся:

- запрос на объединение и предварительную фильтрацию данных
- запрос на расчёт периодов очереди с использованием оконных функций

В решении использовались `LEAD`, `LAG` и `SUM OVER`.

Ограничения

В исходных данных нет информации о фактическом времени прихода покупателей в очередь. Поэтому очередь определяется косвенно, через временной разрыв между операциями.

Из-за этого подход позволяет оценить периоды высокой загрузки кассы, но не показывает фактическую длину очереди.

Возможные улучшения

Проект можно расширить в нескольких направлениях:

- добавить анализ очередей по времени суток
- построить визуализации нагрузки по кассам
- использовать event-based подход при наличии более детальных данных
